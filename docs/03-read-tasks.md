# Phase 2-1：读取任务表

目标：
- 从任务表中提取本轮需要处理的三方库
- 建立后续克隆与适配所需的基础任务数据
- 按多库串行规则确定执行顺序

## 输入

- `libs/porting-tasks-YYYY-MM-DD.xlsx`

## 输出

- 待处理库列表
- 每个库的最小元信息：
  - 库名
  - 仓库地址
  - 版本
  - 是否需要用户审批方案
  - 审批结果
  - 适配状态
  - 编译状态
  - 测试状态
  - 失败原因/备注

## AI 执行步骤

### 1. 找到当天正式任务表

优先读取当天日期对应文件；若不存在，则读取 `libs/` 中最新的正式任务表。

约束：
- 不允许直接读取 `libs/porting-tasks-模板.xlsx`
- 只有 `porting-tasks-YYYY-MM-DD.xlsx` 这类正式任务表才能作为 Phase 2 输入
- 如果只有模板、没有正式任务表，说明还停留在 Phase 1，必须等待用户补完后再继续

### 2. 读取有效行

只处理至少具备以下字段的行：
- 库名
- Git 仓库

推荐执行：

```bash
bash scripts/read-task-sheet.sh --format tsv
```

说明：
- 该脚本会读取正式任务表并只输出有效任务行
- 默认优先读取 `libs/` 中最新的 `porting-tasks-YYYY-MM-DD.xlsx`
- 若宿主已有 `python3 + openpyxl`，脚本直接解析
- 若仅缺 `openpyxl`，脚本会优先尝试自动安装

### 3. 多库排序规则

任务表中的多库执行顺序固定如下：
- 先处理 `是否需要用户审批方案=否` 的库
- 再处理 `是` 或留空的库
- 同组内按表格顺序串行

默认值规则：
- `是否需要用户审批方案` 留空，默认 `是`
- `审批结果` 留空时：
  - 若无需审批，则写作 `不需要审批`
  - 若需要审批，则写作 `待审批`

### 4. 当前轮次与审批流

多库模式下：
- 不需要审批的库，可以直接继续到 Phase 6
- 需要审批的库，先统一推进到 Phase 3
- 等用户集中审批后，再继续进入 Phase 4/5/6

### 5. 规范化字段

建议统一成：
- `lib_name`
- `repo_url`
- `version`
- `approval_required`
- `approval_result`
- `adaptation_status`
- `build_status`
- `test_status`
- `note`

### 6. 标记代码来源模型

AI 应在内部区分两类后续路径：
- `libs/<库名>/`：给 Phase 3/4 分析和 fallback 用
- `HPKBUILD recipe`：给 Phase 5 的 `lycium` 用

## 处理规则

- `版本` 为空：允许继续，后续按仓库默认分支处理
- `适配状态`、`编译状态`、`测试状态` 为空：默认视为 `待处理`
- `失败原因/备注` 仅作为辅助信息，不阻塞流程
- 缺少 `库名` 或 `Git 仓库`：才需要用户补全

补充规则：
- 任务表为空，或只有表头没有有效行：中断并提示用户补全，不继续进入代码拉取
