# Phase 3：业务代码适配方案分析

目标：
- 分析三方库源码中的 HarmonyOS 业务适配点
- 输出业务代码适配方案报告

注意：
- 本阶段不负责输出编译构建方案
- 本阶段不讨论 `lycium` 或 fallback

## 输入

- `libs/<库名>/`
- 任务表中的库信息

## 输出

- `reports/<库名>-adaptation-plan.md`

## 分析范围

重点关注这些内容：
- 平台宏判断
- 平台相关头文件
- HarmonyOS 不兼容的系统调用
- 需要替换的系统接口
- 日志、线程、文件系统、网络、安全相关接口
- 其他会影响运行在 HarmonyOS 上的业务代码差异

## 不属于本阶段的内容

- `lycium` 是否可编
- 原生 fallback 命令怎么写
- `build.sh` 怎么生成
- 编译时需要哪些二进制产物

这些都属于 Phase 5。

## AI 执行步骤

### 1. 扫描源码结构

记录：
- 主要目录
- 核心模块
- 入口文件
- 平台相关文件分布

### 2. 识别平台相关代码

重点关注：
- `#ifdef _WIN32`
- `#ifdef __linux__`
- `#ifdef __APPLE__`
- 平台专用头文件
- 直接系统调用

### 3. 给出 HarmonyOS 适配建议

每个适配点至少说明：
- 文件路径
- 影响范围
- 需要替换或新增的逻辑
- 适配原因

### 4. 输出方案报告

建议报告结构：

```markdown
# <库名> 业务代码适配方案

## 1. 项目结构概览
## 2. 平台相关代码识别
## 3. HarmonyOS 业务适配点
## 4. 建议修改清单
## 5. 风险与假设
```
