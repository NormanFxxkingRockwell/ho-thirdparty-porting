# Phase 3：业务代码适配方案分析

目标：
- 识别三方库中的 HarmonyOS 业务适配点
- 形成业务代码适配方案
- 为 Phase 5 提供最小构建交接摘要
- 额外整理后续测试入口和建议

输出物：
- `reports/<库名>/adaptation-plan.md`

## 必须覆盖的内容

### 1. 项目结构概览

- 核心源码目录
- 公开头文件目录
- 示例程序、工具程序、测试程序目录

### 2. 平台相关代码识别

- 平台宏
- 系统调用
- 平台相关头文件
- 线程、文件、网络、时间、进程等能力调用

### 3. HarmonyOS 业务适配点

- 哪些接口需要替换成 HarmonyOS 对应实现
- 哪些平台判断需要新增或调整
- 哪些逻辑暂时可以跳过，优先让流程走通

### 4. 可复用测试入口与指导

本节是给后续测试 phase 直接使用的，不可省略。

至少整理：
- 上游已有的 test program
- 若无合适 test program，上游可复用的 CLI
- 推荐优先使用的运行命令
- 样例输入、参数或测试数据
- 若无现成入口，明确记录“无测试用例”

### 5. 给 Phase 5 的最小交接摘要

只保留最少必要字段：
- 构建系统类型
- 是否发现现成 `HPKBUILD`
- 是否更适合优先尝试 `lycium`
- 是否预计需要 fallback
- 已知高风险依赖或构建障碍

说明：
- 这里不是完整构建方案
- 这里只是避免 Phase 5 重新摸底

## 多库模式下的审批规则

如果该库的 `是否需要用户审批方案=否`：
- Phase 3 完成后不进入 STOP 2
- 可直接继续进入 Phase 4
- 同时应将任务表中的 `审批结果` 写成 `不需要审批`

如果该库的 `是否需要用户审批方案=是` 或留空：
- Phase 3 完成后将 `审批结果` 写成 `待审批`
- 输出 `reports/<库名>/adaptation-plan.md`
- 等待用户后续集中审批

如果用户将 `审批结果` 设为 `不通过`：
- 该库回到 Phase 3
- 重写或更新 `reports/<库名>/adaptation-plan.md`
- 再次等待审批

## STOP 2 前的要求

- 必须生成 `reports/<库名>/adaptation-plan.md`
- 必须明确是否存在可复用 test program，以及若无 test program 时是否存在可复用 CLI
- 必须明确若无 test program 时，当前库是否无现成测试用例
- 对需要审批的库，到达 STOP 2 后只等待用户审批，不得自行进入 Phase 4
