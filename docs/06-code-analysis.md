# Phase 3: 代码分析与适配方案

## 前置准备

### docs_index MCP 使用方式

在开始本 Phase 前，先了解 docs_index MCP 的用法：

1. 读取 docs_index 仓库的 README.md
2. 了解如何调用 docs_index 检索 HarmonyOS API 文档
3. 后续每个步骤中，先通过 docs_index 检索相关领域 API，再进行分析

**核心原则**：docs_index 不是独立步骤，而是贯穿所有步骤的基础能力。

---

## 输入/输出

**输入**：
- 已克隆的库源码（`libs/<库名>/`）
- 任务表格信息（库名、版本、Git 仓库）

**输出**：
- `reports/<库名>-adaptation-plan.md` - 适配方案（待用户审核）

---

## 步骤 1: 扫描项目结构

**执行**：
1. 遍历 `libs/<库名>/` 目录
2. 记录目录层级和文件类型分布
3. 识别入口文件（`main.c`, `*.h`, `CMakeLists.txt`, `Makefile`, `configure.ac` 等）

**输出到 `reports/<库名>-adaptation-plan.md`**：

```markdown
## 1. 项目结构

**目录组织**：
（描述项目的目录层级和文件分布）

**关键文件**：
- 入口文件：xxx
- 配置文件：xxx
- 核心模块：xxx

**项目规模**：
- 文件数：N
- 代码行数：N
- 语言：C/C++/混合
```

---

## 步骤 2: 识别库的用途

**执行**：
1. 阅读 `README.md`, `package.json`, `CMakeLists.txt`
2. 通过 docs_index 检索该库的官方文档（如有）
3. 分析核心 API 和功能模块

**输出到适配方案**：

```markdown
## 2. 库的用途

**类型**：{网络库/日志库/加密库/数据结构库/...}

**核心功能**：[功能 1, 功能 2, ...]

**主要 API**：[API1, API2, ...]

**涉及的 HarmonyOS 领域**：{logging/network/security/filesystem/...}
```

**作用**：为步骤 3 的平台相关代码识别做铺垫。

---

## 步骤 3: 识别平台相关代码（同步写入适配点）

**执行**：
1. 使用子 agent 阅读源代码文件
2. 基于 LLM 分析判断平台相关代码（不依赖固定模式匹配）
3. 判断标准（供 AI 参考，非硬性规则）：
   - 条件编译：`#ifdef _WIN32`, `#ifdef __linux__`, `#ifdef __APPLE__` 等
   - 系统 API 调用：非标准 C/C++ 的系统调用
   - 平台特定头文件：`<windows.h>`, `<unistd.h>`, `<pthread.h>` 等
   - 其他平台敏感代码

**关键原则**：
- ✅ 灵活判断，基于语义分析
- ✅ 使用子 agent 阅读代码理解意图
- ❌ 不依赖固定关键词/正则匹配

**同步写入适配点清单**：

每发现一个适配点，立即写入适配方案的步骤 4 中，格式如下：

```markdown
| 文件路径 | 代码行数 | 修改详情 | 修改原因 |
|----------|----------|----------|----------|
| src/xxx.c | 25-30 | 添加 `#ifdef __OHOS__` 分支 | 缺少鸿蒙平台检测 |
| src/yyy.c | 42-55 | 替换 `syslog()` 为 `HiLog` | POSIX API 需鸿蒙化 |
```

**输出到适配方案**：

```markdown
## 3. 平台相关代码识别

**条件编译**：
（列出条件编译宏及位置）

**系统 API 调用**：
（识别的系统 API 及位置）

**适配点清单**：
（表格形式，每发现一个写入一个）
```

---

## 步骤 4: 提交用户审核

**执行**：
1. 汇总步骤 1-3 的所有输出
2. 检查适配点清单是否完整
3. 提交用户审核

**发送消息**：

```
✅ 适配方案已生成：`reports/<库名>-adaptation-plan.md`

请审核：
1. 所有适配点是否完整
2. 修改指令是否准确
3. 实施顺序是否合理

如无异议请回复："批准方案"
```
