# Phase 4-2：业务适配报告

输出物：
- `reports/<库名>/adaptation-report.md`

目标：
- 记录 Phase 4 实际实施的业务适配修改
- 标记与方案不一致的地方
- 把测试入口、关键 API 和样例输入交接给 Phase 5

## 建议结构

```markdown
# <库名> 业务适配报告

## 1. 输入方案
## 2. 已实施修改
## 3. 与方案的差异
## 4. 遗留业务适配问题
## 5. 测试入口与使用建议
## 6. 交接给 Phase 5 的说明
```

## 关键要求

### 1. 区分业务适配修改与编译驱动修改

- 业务适配修改写在本报告中
- 编译驱动型修改写在 `reports/<库名>/build-report.md`

### 2. 测试入口与使用建议必须补全

至少包括：
- 优先测试入口路径
- 该测试入口应优先是上游自带、可独立运行的程序，而不是按目录名硬编码判断
- 若无合适测试入口，优先 CLI 能力校验路径
- 若没有合适测试入口，明确记录“无测试用例”
- 关键 API、参数、样例输入
- 推荐执行命令

### 3. Phase 5 直接复用本报告

Phase 5 不应重新猜测试入口。
应优先复用本报告中已经整理好的：
- 上游自带、可独立运行的测试入口
- 候选 CLI 能力校验入口
- 推荐命令
- “无测试用例”的明确结论
